# myCode 阶段 02：工具系统规格说明

## 阶段标识

- 阶段编号：Stage 02
- 阶段名称：工具系统
- 阶段目标：让 OpenAI 系列模型能通过统一工具系统读取文件、写文件、修改文件、执行命令、查找文件和搜索代码内容。

## 背景

myCode 已完成 Stage 01 纯对话 TUI：TUI 通过会话协调层消费统一 LLM 流式事件，协议层分别适配 Anthropic、OpenAI Responses 和 OpenAI Chat Completions。当前系统只能把用户输入发给模型并流式输出 assistant 文本，不能操作本地项目。

本阶段在现有 LLM、protocols、memory 和 session 边界上加入最小可用工具系统。模型在流式响应中发起工具调用时，myCode 能拼接 JSON 参数碎片，按工具名查找注册工具，执行本地动作，并把结构化工具结果写回对话历史。

本阶段只打通 OpenAI Responses 和 OpenAI Chat Completions。Anthropic 暂不实现工具调用，但内部事件和工具抽象应避免写死 OpenAI，以便后续扩展。

## 目标

- 提供统一工具抽象，让每个工具暴露名称、描述、参数 Schema 和执行入口。
- 将工具相关类和实现统一放在 `src/mycode/tool` 包下。
- 首批提供六个核心工具：读文件、写文件、改文件、执行命令、按模式找文件、搜代码内容。
- 提供工具注册中心，集中登记工具，支持按名称查找，并能生成 OpenAI API 可识别的工具定义列表。
- 工具执行带超时和错误处理，失败信息以结构化结果返回。
- OpenAI Responses 和 OpenAI Chat Completions 能解析流式工具调用参数碎片，并产出统一内部工具调用事件。
- 会话协调层能执行一次工具调用，把结果回灌进对话历史，然后停止本轮。

## 功能需求

- F1: 系统提供统一工具抽象。每个工具都能暴露名称、描述、参数 Schema，并提供执行入口。
- F2: 工具相关类和实现统一放在 `src/mycode/tool` 包下，对外由清晰的包入口导出。
- F3: 首批提供六个核心工具：读文件、写文件、改文件、执行命令、按模式找文件、搜代码内容。
- F4: 系统提供工具注册中心，集中登记工具，支持按名称查找，并能生成 OpenAI API 可识别的工具定义列表。
- F5: 工具执行必须有超时和异常捕获。执行失败时返回结构化错误结果，而不是让会话崩溃。
- F6: 改文件工具使用原文唯一匹配替换：匹配不到或匹配多次时必须返回清楚错误，让模型能根据错误重试。
- F7: 文件读写和改文件工具共用一层进程内文本缓存；缓存更新必须加锁，避免并发工具执行时读写状态串扰。
- F8: OpenAI Responses 协议支持流式解析工具调用，能拼接 JSON 参数碎片，并在流结束时产出统一内部工具调用事件。
- F9: OpenAI Chat Completions 协议支持流式解析 `tool_calls` 参数碎片，并在流结束时产出统一内部工具调用事件。
- F10: 会话协调层能识别模型发起的工具调用，执行对应工具，把工具结果以对应 OpenAI 协议可继续理解的历史消息格式写回记忆。
- F11: 本阶段不启动自动 Agent Loop。一次用户请求中，模型如果调用工具，myCode 执行并写回结果后，本轮结束；不会继续调用 LLM 生成第二段回答。
- F12: 如果模型请求不存在的工具、参数不是合法 JSON、参数校验失败、工具执行超时或工具执行异常，系统都要把错误包装成结构化工具结果写回历史。
- F13: TUI 能以可读方式提示工具已执行或工具失败，但不需要做复杂全屏工具面板。

## 非功能需求

- N1: 工具系统边界必须独立于具体 LLM Provider。工具定义、注册、执行结果和错误结构不能依赖 OpenAI 专有字段。
- N2: OpenAI Responses 和 OpenAI Chat 的协议差异只允许留在协议适配层与历史消息转换层中，不能污染工具实现。
- N3: 首批工具只允许在当前工作目录范围内操作文件，避免模型随意读写工作区外路径。
- N4: 文件写入和改写需要使用 UTF-8 文本；二进制文件不作为本阶段目标。
- N5: 命令执行必须有默认超时，并返回退出码、stdout、stderr、是否超时等结构化字段。
- N6: 文件搜索和代码搜索应优先使用 Python 标准库实现，保证测试稳定；可在后续再切换到 `rg` 等外部命令优化性能。
- N7: 自动化测试不得依赖真实 OpenAI API key、真实外部网络或真实用户终端输入。
- N8: 工具调用流式解析必须可单元测试，尤其覆盖 JSON 参数被拆成多个碎片的情况。
- N9: 错误信息要足够具体，但不能泄露敏感配置，例如 API key。
- N10: 保持现有纯对话路径可用。模型没有发起工具调用时，行为应与 Stage 01 基本一致。
- N11: Anthropic 暂不实现工具调用，但新增内部结构时要能容纳未来的 Anthropic tool use 映射。
- N12: 本阶段不引入持久化索引、长期记忆、多 agent 调度或复杂权限确认流程。
- N13: 文件文本缓存必须是工具注册实例持有的对象，不使用模块级可变全局状态；缓存读写和同一路径的读改写必须受锁保护。

## 不做的事

- 不做自动 Agent Loop：工具执行结果回灌后，本轮停止，不自动再次请求模型。
- 不做多工具连环调用，也不做并行工具调用调度；请求层会尽量限制一次只产生一个工具调用。
- 不接入 Anthropic 工具调用，只保留内部事件和工具抽象的扩展空间。
- 不做项目索引、RAG、符号级代码理解或 AST 级编辑。
- 不做二进制文件读取、写入或编辑。
- 不做复杂权限系统、用户逐次确认弹窗或沙箱隔离。本阶段用限制在工作目录内、超时和结构化错误控制风险。
- 不做 Git 自动提交、自动 diff 审批或补丁队列。
- 不做复杂 TUI 工具面板，只输出简洁状态提示。
- 不改变现有配置文件格式；工具系统默认启用，后续如需开关再加配置。
- 不保证命令执行跨平台语义完全一致，只保证在当前 Python 运行环境中通过测试覆盖核心行为。

## 验收标准

- AC1: 六个核心工具都能通过统一接口暴露名称、描述、参数 Schema 和执行入口，并能被注册中心按名称查找。
- AC2: 注册中心能生成 OpenAI Responses 和 OpenAI Chat 都可使用的工具定义列表。
- AC3: 读文件工具能读取工作目录内 UTF-8 文本文件；工作目录外路径返回结构化错误。
- AC4: 写文件工具能写入工作目录内 UTF-8 文本文件，并创建必要父目录；工作目录外路径返回结构化错误。
- AC5: 改文件工具只在原文出现一次时替换成功；出现零次或多次时返回明确结构化错误。
- AC6: 读文件、写文件和改文件共用同一个带锁文本缓存；连续读同一未变化文件可命中缓存，写入或改写后后续读取返回最新内容。
- AC7: 并发执行同一路径的读写或改写时，缓存内容和磁盘内容保持一致，不出现跨请求共享变量串扰。
- AC8: 执行命令工具能返回退出码、stdout、stderr；超时时返回结构化超时结果。
- AC9: 按模式找文件工具能返回匹配路径列表，并限制在工作目录内。
- AC10: 搜代码内容工具能返回包含匹配文本的文件路径、行号和行内容，并限制在工作目录内。
- AC11: OpenAI Responses 流式解析能把拆碎的 function call arguments 拼成完整 JSON，并产出内部工具调用事件。
- AC12: OpenAI Chat 流式解析能把拆碎的 `tool_calls[].function.arguments` 拼成完整 JSON，并产出内部工具调用事件。
- AC13: 会话层收到工具调用后能执行对应工具，并把工具结果写回历史；本轮不继续自动请求模型。
- AC14: 工具不存在、JSON 参数非法、执行超时、执行异常、路径越界、改文件匹配失败等情况都不会让程序崩溃，而是形成结构化错误结果。
- AC15: 模型不调用工具时，原有纯文本流式聊天、多轮记忆、thinking 不入普通历史等 Stage 01 行为仍保持通过。
- AC16: 相关单元测试和端到端 mocked 测试全部通过，不依赖真实网络或真实 API key。
