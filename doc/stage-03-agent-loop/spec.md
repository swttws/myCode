# myCode 阶段 03：Agent Loop 与事件流规格说明

## 阶段标识

- 阶段编号：Stage 03
- 阶段名称：Agent Loop 与事件流
- 阶段目标：让 myCode 从单次工具调用升级为可多轮运行的 ReAct Agent，并通过稳定事件流向 TUI / CLI 暴露执行过程。

## 背景

myCode 已完成 Stage 02 工具系统：OpenAI Responses 和 OpenAI Chat Completions 可以发起单次工具调用，系统能执行工具、把结构化结果写回会话历史，并将简洁状态展示给 TUI。当前限制是：工具结果回灌后，本轮会立即结束，不会继续请求模型生成后续回复，也不支持一轮中多个工具调用的调度。

Stage 03 要把“单次工具调用”升级为稳定的 Agent Loop。系统以 ReAct 范式运行：一轮包含模型调用、响应消费、工具调度、工具结果回填和下一轮判断；当模型不再请求工具、显式结束、达到轮数上限、用户取消或出现不可恢复错误时终止。

本阶段还要把过程以稳定事件流暴露给上层。用户消息、模型 thinking、模型文本、工具调用开始、工具结果、最终回复、错误、取消、暂停等待审批等都作为统一事件输出，让 TUI / CLI 只依赖事件契约消费过程，而不直接理解底层协议细节。

## 目标

- 提供独立的 Agent 循环边界，承载 ReAct 多轮流程，不把循环调度逻辑堆进 TUI / CLI。
- 定义稳定公开的事件流契约，并由统一位置管理事件类型和字段。
- 支持状态机式终止判断：继续、正常终止、错误终止、用户取消、暂停等待审批。
- 支持一轮模型响应中的多个工具调用，按读类并发、写类串行的批次规则执行。
- 支持会话内切换 `plan-only` 模式；该模式下遇到写类工具时暂停等待用户审批，审批只放行当前写工具。
- 支持外部取消与超时；取消后循环尽快停止，并避免未完成工具结果污染后续历史。
- 本章仅使用最小可用 system prompt，不引入复杂提示词组装、完整权限系统、Agent 递归调用或后续章节能力。

## 功能需求

- F1: 系统提供独立 Agent Loop。一次用户请求进入 Agent 后，系统能按 ReAct 流程重复执行“调用模型、消费响应、执行工具、回填工具结果、判断下一步”，直到达到终止条件。
- F2: Agent Loop 相关实现代码放在独立的 `agent` 目录边界下，并通过清晰入口供现有会话层、TUI 和 CLI 使用。
- F3: 系统提供稳定公开的过程事件流。上层能通过事件观察用户消息、模型 thinking、模型文本、工具调用开始、工具结果、最终回复、错误、取消、暂停等待审批等过程。
- F4: 事件类型和事件字段由统一位置管理。TUI / CLI 不直接依赖底层 LLM 协议事件，也不直接拼接工具执行细节。
- F5: Agent 在每轮结束时进行状态判断。继续条件是模型请求了可继续处理的工具调用；终止条件至少包括：模型显式结束、模型没有工具调用、达到最大轮数上限、用户取消、不可恢复错误。
- F6: 达到最大轮数上限时，Agent 产出明确错误事件并结束；该情况不伪装成正常最终回复。
- F7: 一轮模型响应里出现多个工具调用时，Agent 按模型给出的顺序分批执行：连续读类工具组成并发批；写类工具作为顺序边界，单独串行执行；写工具后的读工具进入后续批次。
- F8: 每个工具定义显式声明工具分类，至少区分读类和写类。工具调度器只依据该显式分类做分批，不根据工具名称或参数猜测读写属性。
- F9: Agent 执行工具前后预留拦截点。拦截点能观察工具调用、当前模式和上下文，并能决定放行、拒绝、暂停等待审批或返回结构化拦截结果。本阶段不实现完整权限策略。
- F10: 系统支持会话内切换 `plan-only` 模式。进入该模式后，读类工具允许执行；写类工具默认不直接执行，而是触发暂停等待用户审批。
- F11: `plan-only` 模式下，用户审批写类工具时，只放行当前这一个写工具调用；审批后 `plan-only` 状态仍保持开启，后续写类工具仍需再次审批。
- F12: `plan-only` 模式下，用户拒绝待审批写类工具时，Agent 不执行该工具，而是回填结构化拒绝结果，让模型继续输出可审批计划；用户取消审批时，Agent 产出取消事件并结束本轮。
- F13: `plan-only` 模式的最终输出应是一份可供用户审批的计划；如果模型在该模式下尝试写入，事件流必须让上层能明确展示“等待审批”的状态和待审批工具信息。
- F14: Agent 支持外部取消信号。取消到达后，Agent 尽力取消正在进行的模型请求或工具执行，停止推进后续循环，并避免未回填的工具结果写入 memory。
- F15: Agent 支持超时控制。模型调用、工具执行或整次 Agent 请求达到配置的超时限制时，应产出明确错误事件，并保证状态不进入半完成的继续状态。
- F16: Agent 保持 Stage 02 已有能力可用。OpenAI Responses 和 OpenAI Chat 的工具调用历史仍能被正确回填；模型不调用工具时，纯文本聊天路径仍能正常输出并写入历史。
- F17: 本阶段 TUI / CLI 通过 Agent 事件流展示过程。它们至少能展示文本增量、thinking、工具开始、工具结果、最终回复、错误、取消和等待审批状态；thinking 展示继续受现有配置控制。
- F18: `/clear` 仍能清空普通对话历史、工具调用历史和 Agent 运行状态；`plan-only` 的会话状态也应有明确的清空或复位行为。
- F19: 自动化测试能在不访问真实外部网络、不依赖真实 API key、不需要真实终端交互的情况下验证 Agent Loop、事件流、工具分批、plan-only、取消和超时行为。

## 非功能需求

- N1: Agent Loop 边界必须独立于具体 LLM Provider。OpenAI Responses、OpenAI Chat 和未来 Anthropic 的协议差异只允许留在协议适配层或历史转换层。
- N2: Agent 事件契约应稳定、可测试、可被多个上层入口复用。TUI / CLI 不应依赖 provider-specific 事件字段。
- N3: Agent Loop 相关实现代码必须集中在独立 `agent` 包或目录边界下；现有 `tool`、`llm`、`protocols`、`memory` 只做必要适配，不承担循环主逻辑。
- N4: 工具读写分类必须是工具定义的一部分，并保持向后兼容已有六个核心工具的调用能力。
- N5: 工具分批执行必须保持模型给出的顺序语义。读类工具只在相邻且没有写类边界时并发；写类工具不得与其他工具并发执行。
- N6: 并发读工具执行时，单个工具失败不应导致其他已完成读工具结果丢失；该批结果应以可观测事件逐个输出，并以结构化结果回填。
- N7: 写类工具串行执行时，如果某个写工具失败，本章不做智能回滚；失败结果回填给模型，由后续 ReAct 轮次决定是否修正或终止。
- N8: `plan-only` 模式必须默认保守。写类工具在未获审批前不得执行，也不得提前把成功结果写入历史。
- N9: 等待审批是 Agent 的可恢复状态。系统应能从“用户批准当前工具”恢复继续，也能从“用户拒绝当前工具”回填拒绝结果后继续；用户取消审批时应安全结束本轮。
- N10: 取消和超时必须优先保证状态一致性。已经完整写入 memory 的历史可以保留；尚未完成或尚未确认回填的工具结果不得写入 memory。
- N11: 错误事件必须包含机器可判断的错误类别和人类可读说明，但不得泄露 API key 等敏感配置。
- N12: 本阶段不要求真实并行带来性能收益，但行为必须可观测、可测试；并发读工具的测试可使用 fake 工具或受控延迟工具验证。
- N13: 本阶段保持现有配置格式尽量稳定。必须新增配置时，应限定在 Agent 运行参数，例如最大轮数、超时默认值；不引入复杂权限配置。
- N14: 自动化测试不得依赖真实 OpenAI/Anthropic 网络调用、真实用户审批输入或真实外部命令副作用；相关行为用 fake LLM、fake tool 和 mock TUI/CLI 验证。
- N15: 最小 system prompt 只用于让模型知道当前可用工具和 `plan-only` 约束；复杂 prompt 组装、角色分层、长期上下文压缩留给后续阶段。

## 不做的事

- 不做复杂系统提示词组装。本阶段只提供最小可用 system prompt 或最小运行提示，确保 Agent Loop 能跑通。
- 不做完整权限策略。本阶段只实现工具执行前后的拦截位，以及 `plan-only` 下写类工具的审批暂停；不实现多级权限、路径级策略、命令风险分级或持久化授权。
- 不把 Agent 当工具递归调用，不做子任务委派、多 Agent 调度或 Agent-to-Agent 协作。
- 不引入项目索引、RAG、长期记忆、代码符号图谱或上下文压缩。
- 不实现复杂 TUI 全屏面板。TUI / CLI 只需要能基于事件流展示关键状态，并能处理 `plan-only` 切换与写工具审批。
- 不实现工具执行失败后的自动回滚。写工具失败时仅回填结构化失败结果，让后续模型轮次决定如何处理。
- 不改变 Stage 02 已有工具的业务语义。读文件、写文件、改文件、执行命令、查找文件、搜索代码仍保持原有能力和边界，只增加必要的工具分类元信息。
- 不强制接入 Anthropic 工具调用。本阶段继续以 OpenAI Responses 和 OpenAI Chat 为主要工具调用路径，但 Agent 边界不能写死 OpenAI。
- 不做真实网络或真实 API key 依赖的验收。所有测试通过 fake LLM、mocked stream 和 fake tool 覆盖。
- 不做其他后续章节能力；凡未在本 spec 明确列出的 Agent 扩展能力都留到后续阶段。

## 验收标准

- AC1: 给定一个 fake LLM 先请求读工具、再基于工具结果输出文本，Agent 能完成至少两轮 ReAct，并产出最终回复事件。
- AC2: 给定模型不请求工具的普通文本响应，Agent 能流式产出模型文本、最终回复事件，并将 user / assistant 历史写入 memory。
- AC3: 给定模型显式结束或响应完成且没有工具调用，Agent 能正常终止，不再额外请求 LLM。
- AC4: 给定模型持续请求工具直到超过最大轮数，Agent 产出包含 `max_rounds_exceeded` 类别的错误事件，并结束循环。
- AC5: Agent 事件流包含统一管理的事件类型，至少覆盖：用户消息、thinking、模型文本、工具调用开始、工具结果、最终回复、错误、取消、暂停等待审批。
- AC6: TUI / CLI 只通过 Agent 事件流展示过程；测试中不需要直接消费 provider-specific LLM 事件或自行执行工具。
- AC7: 一轮响应中如果模型按顺序请求“读、读、写、读”，Agent 会并发执行前两个读工具，随后串行执行写工具，最后执行后一个读工具；事件顺序能反映这些批次边界。
- AC8: 每个默认核心工具都显式声明读/写分类；读类包括读文件、查找文件、搜索代码，写类至少包括写文件、改文件、执行命令。
- AC9: 工具分类缺失或非法时，Agent 不猜测分类，而是产出明确错误或拒绝注册/执行。
- AC10: `plan-only` 模式可在会话内开启和关闭；开启后读类工具仍能执行。
- AC11: `plan-only` 模式下遇到写类工具时，Agent 产出暂停等待审批事件，且在审批前不执行该写工具、不写入成功工具结果。
- AC12: 用户批准 `plan-only` 下的当前写工具后，Agent 只放行该工具一次，并继续保持 `plan-only`；后续写工具仍会再次触发审批。
- AC13: 用户拒绝待审批写工具时，Agent 不执行该写工具，回填结构化拒绝结果，并让模型继续输出计划；用户取消审批时，Agent 产出取消事件并结束本轮。
- AC14: 外部取消到达时，Agent 产出取消事件并停止推进；未完成或未审批的工具结果不会写入 memory。
- AC15: 模型调用、工具执行或整次 Agent 请求超时时，Agent 产出带机器可读类别的错误事件，并保持 memory 中没有半完成工具结果。
- AC16: 工具执行前后拦截点可被测试替换；测试能验证拦截器可以放行、拒绝和暂停等待审批。
- AC17: Stage 02 的 OpenAI Responses 和 OpenAI Chat 工具调用历史格式仍可用；工具结果回填后下一轮 LLM 请求能看到对应历史。
- AC18: `/clear` 后，普通对话历史、工具历史、暂停审批状态和 `plan-only` 状态按 spec/plan 确定的规则复位。
- AC19: 相关单元测试和 mocked 端到端测试全部通过，且不依赖真实外部网络、真实 API key 或真实终端交互。
- AC20: README 或阶段文档说明 Stage 03 已支持 Agent Loop、事件流、工具分批执行、plan-only 审批和取消/超时边界，同时明确列出本阶段不做的能力。
