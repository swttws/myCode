# myCode Stage 05：纵深权限与安全检查规格说明

## 阶段标识

- 阶段：Stage 05
- 名称：纵深权限与安全检查
- 目标：在所有工具执行前建立统一、可解释、可配置且默认安全的权限决策链，并由文件工具的路径沙箱提供独立的执行期保护。

## 背景

当前工具系统已经具备两个安全基础：文件工具通过工作区路径守卫限制访问范围，`plan-only` 模式会在写工具执行前请求一次性审批。现有能力仍有以下缺口：

- 普通模式下没有统一的危险命令检查和权限规则。
- 审批只支持当前调用，不能形成会话级或项目级授权。
- 没有严格、默认、放行等整体权限档位。
- 路径守卫、审批和 Agent 拦截接口分散在不同领域包中，权限职责不集中。
- 权限拒绝缺少稳定的原因、来源和脱敏诊断信息。
- `run_command` 在工作区中启动 shell，但 shell 子进程仍能访问工作区外；当前路径守卫不能被解释为操作系统级进程沙箱。

本阶段需要在现有 Agent Loop、工具调度、审批事件和文件工具基础上增加统一权限领域，同时保持供应商协议、conversation memory 和工具结果历史语义不变。

## 威胁模型

本阶段防御以下风险：

- 模型误判导致的破坏性工具调用。
- 提示注入诱导模型执行高风险命令或访问越界路径。
- 未明确授权的写入、删除、命令执行和其他高风险工具操作。

本阶段信任 myCode 进程、本地用户、内置工具实现，以及由本地用户持有的全局配置、本地项目配置和通过 HITL 创建的授权，不防御恶意插件、恶意工具实现或已经控制本机账户的攻击者。仓库随附的项目权限文件属于工作区内容，不视为可信授权来源；它只能收紧权限，不能创建 `ALLOW` 或降低有效权限档位。

## 目标

- 所有工具调用在执行前经过统一权限决策。
- 提供不可由运行时审批绕过的安全底线。
- 支持用户全局、项目和会话三层规则，并保证确定性优先级。
- 支持严格、默认、放行三种权限档位。
- 对需要人工判断的操作支持本次、会话和项目三种允许范围。
- 保证内置文件工具不能访问工作区外路径。
- 对危险 shell 命令执行高置信度硬拒绝或风险审批。
- 对权限决定提供中文、可解释且不泄露凭据的用户提示和诊断信息。
- 将权限领域代码集中在独立的 `permission` 包中。

## 功能需求

### F1：统一权限决策入口

每个合法工具调用在进入工具执行器前必须经过同一个权限决策入口。权限入口接收工具名称、工具读写分类、结构化参数、工作区、会话状态和当前 `plan-only` 状态，返回一个确定的权限决定。

权限决定包括：

- `ALLOW`：允许执行。
- `DENY`：按配置或安全失败拒绝执行。
- `ASK`：需要用户审批。
- `FORBIDDEN`：命中不可绕过的内置安全底线。

权限决定同时提供稳定原因码、面向用户的中文摘要、命中规则及作用域等可解释元数据。未知工具、非法 JSON 参数或不符合工具参数契约的调用在权限判断前失败，不得进入执行器。

### F2：权限领域包

权限模型、规则匹配、策略编排、危险命令分析、路径守卫、审批类型、规则存储和 Agent 拦截适配均归属于独立的 `permission` 包。

Agent 领域只负责编排轮次和发布事件，工具领域只负责工具定义与执行，TUI 只展示权限信息并收集用户选择。顶层配置负责定位配置来源，权限配置内容的解析和校验由权限领域负责。

### F3：不可绕过的安全底线

内置安全底线在普通规则和权限档位之外生效。命中 `FORBIDDEN` 的调用直接拒绝，不提供本次、会话或项目授权选项，也不能被 `permissive`、会话规则、项目规则或用户全局规则覆盖。

首期安全底线至少覆盖：

- 递归删除工作区根、用户主目录、文件系统根或磁盘根。
- 格式化磁盘、覆盖原始块设备等不可逆系统破坏。
- 在同一命令链中下载远程内容并直接交给 shell、解释器、`eval` 或等效执行入口。
- 明确针对系统关键目录的大范围删除或权限破坏。

安全底线使用保守的高置信度规则。无法可靠理解但没有高置信度命中安全底线的复杂命令进入 `ASK`，不得被误判为普通安全调用。

### F4：高风险操作分类

系统对具有合理用途但风险较高的操作产生内置 `ASK`。首期至少包括：

- 删除工作区内的文件或子目录。
- `git clean`、`git reset --hard` 等可能丢失工作区内容的版本控制操作。
- 软件包安装和外部网络上传下载。
- 权限修改、提权、服务管理和进程管理。
- 命令分析器无法可靠理解的复杂 shell 结构或可疑混淆。

内置高风险分类不由权限档位的未命中兜底结果覆盖。用户之前创建的精确允许规则可以满足同一高风险调用的授权；`plan-only` 约束仍可要求再次审批。

### F5：分层规则来源

系统支持三个规则作用域，优先级从高到低为：

1. 当前会话临时规则。
2. 当前项目规则。
3. 用户全局默认规则。

内置 `FORBIDDEN` 安全底线始终位于三层普通规则之上。系统只在最高优先级且存在匹配项的作用域中选择结果，不合并较低作用域的匹配项。因此会话普通规则可以覆盖项目普通规则，但不能覆盖安全底线。

用户全局规则从 `~/.mycode/permissions.yaml` 加载；会话规则只保存在内存中。项目作用域包含两个子来源，优先级从高到低为：

1. 用户拥有的本地项目授权：保存在用户目录下，并按规范化工作区标识隔离。该来源可信，可以包含 `ALLOW`、`DENY` 和 `ASK`。
2. 仓库随附的项目策略：从工作区根目录的 `mycode.permissions.yaml` 加载。该来源不可信，只能包含 `DENY` 和 `ASK`，不能设置权限档位。

本地项目授权可以覆盖仓库项目策略中的普通 `DENY` 或 `ASK`，因为它代表本地用户针对当前工作区作出的显式授权；两者都不能覆盖 `FORBIDDEN`。仓库项目策略即使来自恶意仓库，也最多增加拒绝或审批，不能扩大工具权限。

### F6：规则表达与确定性冲突处理

用户全局配置和用户拥有的本地项目授权支持 `ALLOW`、`DENY` 和 `ASK`；仓库随附的项目策略只支持 `DENY` 和 `ASK`。所有规则具有唯一规则 ID、工具匹配条件和可选参数匹配条件。任何配置来源都不能声明或覆盖 `FORBIDDEN`。

首期规则匹配要求：

- 工具条件支持精确工具名和通配工具。
- 字符串参数支持 glob 模式，不支持正则表达式。
- 数字和布尔参数执行精确匹配。
- 路径在匹配前规范化为工作区相对路径并统一使用 `/` 分隔符。
- 命令在匹配前仅规范化首尾空白和连续空白，不改变引号、转义和参数含义。

选择最高优先级作用域后，如果命中项目作用域，先按 F5 选择最高优先级且存在匹配项的项目子来源，再在该来源内计算规则具体度：精确工具优先于通配工具；约束参数更多者优先；精确参数值优先于通配值。具体度相同且结果冲突时使用 `DENY > ASK > ALLOW`。完全相同的匹配条件具有不同结果时，静态配置在加载阶段报告冲突；其他模式重叠在运行时按上述规则确定结果，不依赖声明顺序。

### F7：权限档位

系统支持三种权限档位。档位只决定没有普通规则和内置高风险分类命中时的默认结果：

- `strict`：只有显式 `ALLOW` 可以直接执行，其他未命中调用进入 `ASK`。
- `default`：普通读工具默认允许，写工具和命令工具默认进入 `ASK`。
- `permissive`：普通未命中调用默认允许。

三种档位都不能覆盖 `DENY` 或 `FORBIDDEN`。有效档位按会话设置、用户拥有的本地项目配置、用户全局配置、内置 `default` 的顺序选择。仓库随附的项目策略不能设置档位，避免恶意仓库将模式改为更宽松的 `permissive`。

系统提供会话命令显示和修改有效档位。`/permission` 显示当前档位及来源；`/permission strict|default|permissive` 设置会话级档位；`/clear` 清除会话规则和会话级档位，恢复本地项目、用户全局或内置默认值。

### F8：人在回路审批

结果为 `ASK` 时，系统向用户展示工具名称、脱敏后的安全相关参数、中文风险原因、规则来源和当前权限档位，并支持以下选择：

- 本次允许：只允许当前工具调用。
- 本会话允许：增加一条当前会话精确允许规则，然后允许当前调用。
- 当前项目永久允许：向用户目录下按当前工作区隔离的本地项目授权文件原子写入一条精确允许规则，写入成功后允许当前调用。
- 拒绝：不执行工具，将结构化拒绝结果回填给模型并继续当前 turn。
- 取消：不执行工具并终止当前 turn。

审批自动生成的规则只包含当前调用的安全相关参数。文件工具记录规范化后的精确路径；命令工具记录规范化后的精确命令；文件正文、超时等与权限无关的参数不得写入规则。

审批不能创建用户全局规则，也不能向仓库随附的项目策略写入 `ALLOW`。非交互环境没有审批提供者时，`ASK` 按拒绝处理。本地项目授权写入失败时不得执行当前调用，也不得暗示授权已经持久化。

### F9：`plan-only` 兼容与叠加

现有 `plan-only` 作为独立约束叠加在普通权限决定之上。开启时，写工具的最终决定至少为 `ASK`，已有会话或项目允许规则不能绕过该约束；`DENY` 和 `FORBIDDEN` 保持更严格的结果。

`plan-only` 审批只展示本次允许、拒绝和取消，因为持久授权不能绕过当前模式。关闭 `plan-only` 后，原有普通规则继续生效。现有 `/plan-only` 命令及一次性审批行为保持兼容。

### F10：文件工具路径沙箱

内置文件读取、写入、编辑、查找和搜索工具只能在当前工作区根目录内活动。首期不允许任何配置或授权把路径沙箱扩展到工作区外；普通规则仍可在工作区内部按既定优先级允许、询问或拒绝具体路径。

路径沙箱必须满足：

- 相对路径基于工作区解析，绝对路径只有位于工作区内时才可使用。
- 拒绝父目录跳转、磁盘或文件系统根切换以及其他工作区逃逸。
- 检查已存在目标、符号链接和已存在父目录解析后的真实位置。
- 在规则匹配前和实际文件操作前分别检查路径。
- 查找和搜索工具同时检查起始目录及每个实际读取的候选文件。
- 正确处理 Windows 盘符大小写、UNC 路径和平台分隔符差异。

路径无法规范化、链接检查失败或边界关系无法确认时必须拒绝操作。

### F11：危险命令分析

危险命令分析不能只依赖原始字符串的简单子串匹配。系统需要识别当前平台默认 shell 的命令结构和常见连接操作，并分析显式启动的 `cmd`、PowerShell、`sh`、`bash` 等常见嵌套命令。

分析器需要区分高置信度 `FORBIDDEN`、高风险 `ASK` 和未发现内置风险三类结果，并提供稳定分类原因。无法可靠解析、存在可疑混淆或使用未知 shell 启动器时降级为 `ASK`。本阶段不承诺完整证明任意 shell 字符串安全。

### F12：拒绝结果与执行顺序

只有最终结果为 `ALLOW` 或用户批准当前 `ASK` 后，工具调用才能进入工具执行器。`DENY`、`FORBIDDEN`、审批拒绝和权限组件失败均不得调用真实工具。

权限拒绝作为结构化工具结果回填给模型，使模型可以调整方案；回填内容包含稳定原因码和安全的中文摘要，但不包含完整策略配置、黑名单表达式或内部异常堆栈。

多个并发读工具同时需要审批时，审批请求串行展示，避免 TUI 同时等待多个交互输入。工具通过权限检查后自身执行失败时，继续使用现有工具结果语义，不伪装成权限拒绝。

### F13：配置加载与生命周期

权限配置在启动时加载和完整校验。非法档位、非法 effect、非法模式、重复规则 ID、完全相同条件的冲突结果和其他结构错误必须阻止启动，并指出配置来源及安全的规则位置。仓库随附的项目策略出现 `ALLOW` 或档位设置时同样视为非法配置，不得静默忽略或生效。

HITL 新增的会话和项目规则立即进入当前会话的有效规则集。外部手工修改配置后需要重启；首期不监听文件变化。

本地项目授权持久化必须避免部分写入。持久化失败时保留原文件，并向用户显示中文错误提示。HITL 不修改仓库工作区内的项目策略文件。

### F14：可解释诊断与中文提示

每次权限判定提供以下结构化诊断字段：工具调用 ID、工具名、最终决定、原因码、有效权限档位、命中规则 ID 和作用域、内置风险分类，以及用户审批结果和授权范围。

所有面向用户的权限提示、审批选项、拒绝原因、配置错误和持久化错误使用中文。稳定机器原因码可以使用英文标识，但必须有对应中文提示。

日志不得记录文件正文、完整工具结果或未经脱敏的安全相关参数。审批显示和日志对 `api_key`、`token`、`password`、`secret`、URL 凭据、常见命令行凭据参数及环境变量赋值执行脱敏，并限制诊断值长度。审批界面保留命令结构，只隐藏敏感值。

### F15：关键路径中文注释

权限实现的关键路径必须包含简洁中文注释，说明无法仅从代码表达清楚的安全意图和失败策略。至少覆盖权限优先级与冲突处理、`FORBIDDEN` 不可覆盖、危险命令降级策略、路径边界与符号链接检查、审批持久化失败不执行，以及 `plan-only` 叠加规则。

注释解释“为什么需要此安全约束”，不得逐行复述代码，也不得用注释替代清晰的类型、命名和模块边界。

### F16：现有行为兼容

除本规格明确改变的权限判断外，现有 Agent Loop、多轮工具调用、读工具并发与写工具串行调度、conversation memory、工具结果历史、取消、超时、TUI 事件和供应商协议行为保持不变。

已有文件工具的工作区限制继续生效，但路径守卫的领域归属迁移到 `permission` 包。已有一次性 `plan-only` 审批升级到统一权限流程后，用户可见语义保持兼容。

## 非功能需求

- N1：所有权限领域实现集中在 `permission` 包中；Agent、工具和 TUI 不得复制规则优先级或危险命令判断逻辑。
- N2：相同工具调用、配置、会话状态和工作区上下文必须产生相同权限决定，不受规则声明顺序影响。
- N3：权限入口、命令分析、规则匹配、路径检查和持久化任一环节失败时必须 fail-closed，不得降级为允许。
- N4：安全底线和路径沙箱必须独立于模型提示词；即使模型忽略安全提示，执行前检查仍然生效。
- N5：诊断和日志不得泄露已识别凭据、文件正文、完整工具结果或内部策略细节。
- N6：仓库随附的项目内容不得成为可信授权来源；只有本地用户拥有的配置和 HITL 授权可以扩大普通权限。
- N7：路径行为需要兼容项目支持的平台，并对 Windows 特有路径语义提供覆盖。
- N8：自动化测试不得执行真实危险命令、修改真实用户权限配置、访问真实网络或依赖真实 API key。
- N9：权限判断不能改变现有工具超时、取消和结果序列化契约。
- N10：所有面向用户的权限提示使用中文，权限关键安全路径使用有实质信息的中文注释。
- N11：首期规则和内置风险集合保持有限、明确和可测试，不通过通用脚本或任意表达式扩展策略。

## 不做的事情

- 不实现操作系统级进程、容器或虚拟机沙箱。
- 不保证 shell 子进程只能访问工作区内文件。
- 不防御恶意插件、恶意工具实现、本地恶意用户或已经获得本机账户控制权的攻击者。
- 不完整解析或形式化证明任意 shell 字符串安全。
- 不实现网络隔离、写操作事务或工具失败后的自动回滚。
- 不支持正则权限规则、任意策略脚本或外部策略引擎。
- 不提供图形化权限配置界面、配置热加载或持久化审计数据库。
- 不通过 HITL 创建或修改用户全局规则。
- 不允许仓库随附的项目策略创建 `ALLOW`、设置权限档位或接收 HITL 持久授权。
- 不允许项目配置扩展文件工具到工作区外目录。

## 验收标准

- AC1：所有工具调用在真实工具执行前经过统一权限入口；未知工具和非法参数不会进入执行器。（覆盖 F1）
- AC2：权限模型、规则、审批、命令分析、路径守卫、存储和拦截适配均位于独立 `permission` 包，Agent、工具和 TUI 保持规定职责。（覆盖 F2）
- AC3：删除工作区根、删除系统根、磁盘破坏和同一命令链下载即执行的代表性命令均产生不可审批的 `FORBIDDEN`，三种档位和所有普通规则都不能覆盖。（覆盖 F3）
- AC4：删除工作区子目录、破坏性 Git 操作、软件包安装、网络访问、权限或服务操作以及不可可靠解析的命令进入 `ASK`；精确授权后可按授权范围生效。（覆盖 F4、F11）
- AC5：会话、项目和用户全局规则按既定优先级生效；项目作用域内，本地用户项目授权优先于仓库项目策略；会话和本地项目规则可以覆盖较低层普通规则，但不能覆盖安全底线。（覆盖 F5）
- AC6：仓库随附的项目策略包含 `ALLOW` 或档位设置时启动失败；只包含 `DENY/ASK` 时可以收紧权限，但不能扩大权限。（覆盖 F5、F6、F7、F13）
- AC7：规则测试覆盖精确工具、通配工具、glob 字符串、标量精确匹配、路径规范化、具体度排序和 `DENY > ASK > ALLOW`，结果不受声明顺序影响。（覆盖 F6）
- AC8：`strict`、`default`、`permissive` 对未命中调用产生规定结果，且 `permissive` 不能绕过普通 `DENY`、内置高风险分类或 `FORBIDDEN`。（覆盖 F7）
- AC9：`/permission` 能用中文显示有效档位和来源，三种设置命令立即影响当前会话，`/clear` 恢复本地项目、全局或内置默认值并清除会话规则。（覆盖 F7）
- AC10：审批测试覆盖本次允许、本会话允许、当前项目永久允许、拒绝和取消；自动生成规则不包含文件正文和超时等无关参数。（覆盖 F8）
- AC11：永久授权只写用户目录下按工作区隔离的本地项目授权文件，不修改仓库内容；原子写入成功后当前调用才执行，写入失败时原文件和当前工具均不受影响并显示中文错误。（覆盖 F8、F13）
- AC12：没有审批提供者时 `ASK` 不执行工具；拒绝继续当前 turn，取消终止当前 turn。（覆盖 F8）
- AC13：`plan-only` 开启时写工具至少进入 `ASK`，只显示本次允许、拒绝和取消；普通 `DENY/FORBIDDEN` 保持更严格结果，关闭后普通规则继续生效。（覆盖 F9）
- AC14：文件工具测试证明 `..`、工作区外绝对路径、符号链接逃逸、越界查找根和越界候选文件均无法访问；合法工作区路径继续工作。（覆盖 F10）
- AC15：危险命令分析覆盖当前平台默认 shell、常见嵌套 shell、连接操作、可疑混淆和未知 shell，解析不确定时不会返回普通安全结果。（覆盖 F11）
- AC16：`DENY`、`FORBIDDEN`、审批拒绝和权限内部失败从不调用 fake executor；拒绝结果以安全的结构化内容回填模型。（覆盖 F12）
- AC17：并发读调用需要审批时按顺序展示；工具获准后的执行失败继续保留原工具结果类型和错误语义。（覆盖 F12）
- AC18：非法权限配置在启动前失败并定位到配置来源；HITL 新增规则立即生效；外部配置不要求热加载。（覆盖 F13）
- AC19：权限诊断包含决定、原因、档位、规则和审批元数据；日志及审批摘要对 fixture 中的 API key、token、密码、URL 凭据和命令行凭据完成脱敏，用户提示均为中文。（覆盖 F14）
- AC20：代码审查确认权限优先级、不可覆盖安全底线、命令降级、路径边界、持久化失败和 `plan-only` 叠加等关键路径具有简洁中文注释，且注释说明安全理由而非复述代码。（覆盖 F15）
- AC21：现有 Agent Loop、工具、memory、session、TUI、协议、取消和超时回归测试继续通过。（覆盖 F16）
- AC22：端到端测试使用 fake LLM、临时工作区和 fake command executor，走通普通允许、风险审批、永久授权、恶意仓库项目策略、硬拒绝、路径越界、`plan-only` 和 `/clear`，不执行真实危险命令或访问网络。（覆盖 F1-F16）
