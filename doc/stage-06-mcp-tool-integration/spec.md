# myCode Stage 06: MCP 远端工具接入 Spec

## 背景

myCode 已有本地 Tool 接口、工具注册中心、Agent Loop 与统一权限审批机制，但只能使用进程内实现的工具。用户无法把符合当前 MCP 规范的本地或远程 server 提供的工具接入 Agent。

本期为 myCode 增加 MCP 客户端能力，使用 JSON-RPC 2.0 与多个外部 server 通信，支持本地子进程 stdio 和远程 Streamable HTTP。每个 server 在连接后完成初始化、工具发现和工具调用，并复用已建立的可用连接，避免重复连接。

远端工具将以带 server 名称前缀的唯一名称注册到 myCode。远端工具默认按写工具进入现有权限审批，只有独立 MCP 配置明确指定的工具可按读工具处理。

为避免大量 MCP 工具的完整 schema 占满模型上下文，发现到的 MCP 工具默认延迟注入：模型先通过系统提醒获知可发现的工具名称及其描述；当需要某一工具时，调用工具搜索入口获取其完整定义；被发现的工具从下一轮起作为正常工具提供给模型。

## 目标

- 让 Agent 可无感调用来自多个 MCP server 的工具。
- 支持 stdio 与 Streamable HTTP 两种 MCP 传输。
- 正确处理双向 JSON-RPC 消息、并发请求与按 id 匹配的响应。
- 在连接与工具清单层面缓存可复用状态。
- 保持远端工具受现有权限、审批和 plan-only 机制约束。
- 用独立配置文件声明 MCP server，并支持显式路径和自动发现。

## 功能需求

- F1: 系统从独立 MCP 配置加载多个 server。支持 `--mcp-config` 指定配置文件；未指定时依次发现工作目录的 `mycode.mcp.yaml` 与用户目录的 `~/.mycode/mcp.yaml`。未找到配置文件时，MCP 功能保持禁用，本地工具和聊天功能仍可使用。

- F2: 配置可为每个 server 声明稳定名称、传输方式、连接目标、环境变量和请求超时。stdio server 可声明命令及参数；Streamable HTTP server 可声明 URL 和请求头；敏感值可通过环境变量引用。

- F3: 系统支持 MCP stdio 与 Streamable HTTP 传输，并按当前 MCP 生命周期依次完成初始化握手、初始化完成通知和工具清单发现，再允许工具调用。

- F4: 系统能够同时处理多个未完成的 JSON-RPC 请求，将成功或错误响应按请求 id 关联到原始请求；收到通知时记录其内容；收到 `ping` 请求时成功响应；收到本期未支持的 server 请求时返回 JSON-RPC “方法不支持”错误，不影响其他请求或 server。

- F5: 系统从每个已连接 server 获取工具清单，并把每项工具包装为可由 myCode 调用的远端工具。对 Agent 暴露的工具名必须使用稳定的 server 名称前缀，避免不同 server 的同名工具冲突；调用时仍使用目标 server 的原始工具名和参数。

- F6: 所有远端工具默认按写工具进入既有权限、审批与 plan-only 流程。独立 MCP 配置可逐工具将指定远端工具降为读工具；未显式声明的工具不得绕过写权限默认值。

- F7: 每个 server 的已初始化连接与已发现工具清单可在本进程会话中复用，连续工具调用不得重复建立连接或重复发现。连接中断、超时或协议错误只使该 server 失效；下次需要时系统重新连接和发现，其他 server 与本地工具继续可用。

- F8: 新发现的远端工具默认延迟注入。系统每轮只在系统提醒中列出其可发现名称及对应描述，不将完整 schema 发送给模型。模型可通过始终可用的工具搜索入口请求某个名称的完整定义；系统在本地注册表中解析该工具并将其标记为已发现，使其从下一轮起作为正常工具定义提供给模型。

- F9: 工具搜索入口对不存在、歧义、不可用或所属 server 已失效的工具，返回可供模型理解的结构化失败结果；不得将未成功解析的工具标记为已发现。

- F10: MCP 调用成功时，系统将远端返回的结构化内容转为既有工具结果并提供给 Agent；协议错误、远端错误、超时、断连和无效响应均转为结构化工具失败结果，不得导致 Agent 会话退出。

- F11: 单个配置项无效、连接失败或工具发现失败时，系统向用户和日志报告该 server 的原因，但继续加载其他可用 MCP server 与本地工具。显式指定但不存在或无法解析的 MCP 配置文件应在启动时报告配置错误。

## 非功能需求

- N1: 兼容当前 MCP 规范的 JSON-RPC 2.0、初始化生命周期、stdio 与 Streamable HTTP 传输语义，并协商 server 返回的协议版本和能力。

- N2: 每次连接、初始化、工具发现与工具调用均受该 server 配置的超时约束。超时后停止等待；对于已发送的 MCP 请求，系统尽力发送取消通知，再以结构化失败结束本次工具调用。

- N3: 并发请求、响应、通知、超时和重连不得造成响应错配、挂起等待或状态竞争。一个 server 的故障、慢响应或失效不得阻塞其他 server 或本地工具。

- N4: 进程结束、会话关闭或连接失效时，系统应关闭 HTTP 资源；对于 stdio server，应关闭输入流并在合理等待后终止残留子进程，避免资源泄漏。

- N5: 配置中环境变量解析后的值、HTTP 鉴权头和其他敏感信息不得出现在系统提醒、工具定义、普通错误消息或日志中。诊断信息可包含 server 名称、传输类型和非敏感错误类别。

- N6: MCP 配置必须在建立连接前完成格式和字段校验。server 名称、工具名称与超时值必须符合可预测的命名和数值约束，错误信息应指出配置项位置和原因。

- N7: 延迟工具在模型上下文中暴露其稳定名称以及与该名称对应的远端工具描述，帮助模型判断工具用途；不得暴露参数 schema、环境变量、请求头、远端 URL 或其他配置细节。只有成功发现的工具才发送其完整工具定义。

- N8: 保持现有 Python 3.10 兼容性，并支持项目现有的 Windows 开发与运行环境。

- N9: 自动化测试不得依赖真实 MCP server、外网、真实凭据或真实 API key；应使用可控的本地子进程和 HTTP 模拟服务覆盖正常、并发、超时、断连、协议错误与延迟发现路径。

- N10: MCP 能力未配置或全部 server 不可用时，既有本地工具、权限系统、Agent Loop、TUI 与 LLM 调用行为保持可用。

## 不做的事

- 不支持 MCP resources、prompts、completions、tasks，以及 `roots/list`、sampling、elicitation 等本期范围外的 server 请求；除 `ping` 外的 server 请求统一返回“方法不支持”。

- 不支持已废弃的 MCP HTTP+SSE 传输；远程连接仅支持当前规范的 Streamable HTTP。

- 不实现 OAuth 发现、动态客户端注册、浏览器登录或回调处理；远程鉴权仅使用配置中声明的静态请求头及环境变量引用。

- 不自动监听或刷新 server 的工具清单变更通知。通知会被记录；工具清单在该 server 重新连接和重新发现后更新。

- 不提供跨进程的连接池、持久化工具缓存或后台保活；连接和发现缓存仅在当前 myCode 进程内有效。

- 不提供运行时编辑、热重载或交互式管理 MCP 配置的界面；配置在启动时读取和校验。

- 不在本期实现基于自然语言、关键词或向量检索的工具搜索。模型必须根据提醒中的稳定工具名称请求完整定义。

- 不修改远端工具的业务语义，也不替远端工具建立新的沙箱、网络隔离、重试编排或结果持久化机制。

## 验收标准

- AC1: 使用 `--mcp-config` 指定的有效配置时，系统加载其中的多个 server；未指定时按工作目录和用户目录的顺序发现配置；完全未配置 MCP 时，myCode 仍能启动并使用本地工具。

- AC2: 对受控的 stdio server 和 Streamable HTTP server，系统均能完成 MCP 初始化、初始化完成通知和 `tools/list`，并获得 server 返回的工具清单。

- AC3: 同一 server 有多个并发请求且响应乱序到达时，每个调用均收到与原始请求 id 对应的正确成功或错误结果；任一请求超时后不再无限等待，并产生结构化失败结果。

- AC4: 收到 server 通知时，系统记录该通知但不破坏正在执行的请求；收到 `ping` 时返回成功响应；收到任何本期不支持的 server 请求时返回 JSON-RPC “方法不支持”错误。

- AC5: 两个 server 均提供相同原始工具名时，二者都能以不同的、带 server 前缀的名称出现在 myCode 中；模型调用其中任一个时，参数被发送到正确的 server 和原始工具名。

- AC6: 未在 MCP 配置中声明为读工具的远端工具，在默认权限档位下触发现有写工具审批；配置为读工具的指定远端工具按既有读工具规则处理，且 plan-only 行为保持有效。

- AC7: 对同一可用 server 的连续工具调用复用已初始化连接与工具清单，不重复握手或发现；一个 server 断连、超时或协议错误后，其他 server 和本地工具仍可调用，失效 server 在下次使用时重新连接和发现。

- AC8: 首次向模型提供工具时，未发现的 MCP 工具只以“稳定名称 + 对应描述”出现在系统提醒中，完整 schema 不出现在工具列表。模型搜索某一名称后，系统返回该工具完整定义；下一轮模型请求中仅该已发现工具作为正常工具出现。

- AC9: 搜索不存在、歧义、不可用或 server 已失效的工具时，系统返回结构化失败；这些工具在后续模型请求中仍不出现完整 schema。

- AC10: 远端工具成功返回内容时，Agent 获得既有格式的成功工具结果；远端错误、无效响应、超时和断连均转为结构化失败，Agent 会话和 TUI 不退出。

- AC11: 单个 server 配置错误、连接失败或发现失败时，用户可看到该 server 的非敏感错误，其他有效 server 和本地工具继续可用；显式指定但不存在或格式错误的 MCP 配置文件使启动报告清晰的配置错误。

- AC12: 在连接、初始化、发现和调用超时后，系统按协议尽力取消请求并释放相关 HTTP 或 stdio 资源；退出时不遗留 HTTP 连接或子进程。

- AC13: 自动化测试使用受控的本地 server 或 HTTP 模拟服务，不依赖外网、真实凭据或真实 API key；测试输出、系统提醒和日志不包含已解析的环境变量值或鉴权头。

- AC14: 端到端场景：用户以独立 MCP 配置启动 myCode，两个 server 成功连接；模型从“名称 + 描述”中判断需要一个延迟工具，先搜索其完整定义，再在下一轮调用它；按权限规则完成审批后，用户看到远端工具结果。

- AC15: 端到端边界场景：其中一个配置 server 在运行中失效时，模型仍可完成本地工具或另一 MCP server 的调用；失效 server 被再次需要时，系统重新连接并重新发现，或返回不影响会话的结构化失败。
