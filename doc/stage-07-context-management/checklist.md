# Stage 07 上下文管理 Checklist

> 每一项都通过运行代码、自动化场景或观察外部行为验证；先记录实际证据，再标记通过。

## 配置与预算估算

- [ ] **C1（AC1）上下文窗口配置为必填且阈值可覆盖。** 缺少窗口配置时启动失败并指出字段；合法配置默认采用 8K/12K，显式值能改变触发边界。（验证：分别加载缺失、默认和覆盖配置，观察启动结果与生效阈值。）
- [ ] **C2（AC2）请求前处理顺序固定。** 每次常规模型调用前先完成轻量预防，再估算完整请求，最后判断重量压缩；未重新确认安全的请求不会发送。（验证：运行带顺序记录的请求场景，期望记录为“轻量处理 → 构造/估算 → 重量判断 → 常规调用”。）
- [ ] **C3（AC6）无 usage 时使用完整字符估算。** 混合 ASCII、中文、系统提示、工具定义和历史的请求按约定比例向上取整。（验证：运行固定输入估算场景，对比预先计算的 ASCII/非 ASCII 结果。）
- [ ] **C4（AC6）有效 usage 成为增量锚点。** 下一次估算等于上次 input usage 加字符增减量；中间一轮缺失 usage 不丢锚点，清空会话后锚点重置。（验证：连续运行“有效 usage → 缺失 usage → clear”场景，逐轮对比估算来源和值。）
- [ ] **C5（N8）非法预算组合在启动时被拒绝。** 非正数、2K 预览不小于单项阈值、单项大于批次、批次侵入 13K 安全区均返回明确配置错误。（验证：逐个加载非法配置，期望全部在启动阶段失败且指出对应字段。）

## 轻量预防与归档

- [ ] **C6（AC3）单个超阈值工具结果被完整归档。** 下一次请求只保留最多 2K 的首尾预览、原始大小、截断标记和读取路径，工具调用 ID 不变。（验证：产生一个超过单项阈值的工具结果，捕获下一次模型请求并检查工具调用对应关系。）
- [ ] **C7（AC3/N2）归档内容可无损恢复。** 沿预览路径分段读取后，拼接正文与原工具消息逐字符一致，完整性摘要匹配。（验证：循环读取至 eof，对比原始内容和 SHA-256。）
- [ ] **C8（AC4）同次响应按大小依次归档。** 多个工具结果合计超过批次阈值时，选择顺序按估算大小降序，相同大小保持原消息顺序，每次替换后重新估算。（验证：运行包含不同大小及同大小结果的批次场景，检查归档顺序和每步估算。）
- [ ] **C9（AC4）固定预览无法继续缩小时转交重量判断。** 下一项归档不再降低合计大小时轻量流程终止，不循环归档，随后重量压缩仍能触发。（验证：构造多个接近 2K 的结果，观察轻量停止和后续重量动作。）
- [ ] **C10（AC5）缓存按工作区与会话隔离。** 两个工作区和两个并行会话互相不能读取归档；活动超过 24 小时的会话不会被其他进程清理。（验证：同时创建隔离会话并交叉读取，期望拒绝；运行带活动锁的过期目录清理场景，期望保留。）
- [ ] **C11（AC5）缓存按生命周期清理。** `/clear` 和正常退出删除当前会话目录；超过 24 小时且无活动锁的异常遗留目录在后续启动时删除。（验证：分别执行 clear、正常退出和模拟过期遗留，观察目录存在性。）
- [ ] **C12（AC18）受限读取拒绝越界。** 路径穿越、符号链接逃逸、其他会话路径、未登记用户文件和超过 2K 的读取请求全部失败，不返回目标正文。（验证：运行五类非法读取，检查结构化失败和输出中无敏感正文。）
- [ ] **C13（N9）轻量选择具有确定性和幂等性。** 相同历史与配置重复运行得到相同路径以外的选择、预览和顺序，已归档消息不会再次归档。（验证：对同一 fixture 连续运行两次并比较结果与归档计数。）

## 重量压缩

- [ ] **C14（AC7）自动触发线为窗口减 13K。** 低于边界不产生摘要调用，达到边界时在常规请求前压缩，压缩后请求严格回到边界以下。（验证：分别运行边界减一和正好边界的请求，记录摘要调用数及压缩后估算。）
- [ ] **C15（AC8）`/compact` 可提前手动压缩。** 历史低于自动线但存在旧消息时立即压缩，不创建用户消息或普通聊天请求；无可摘要历史时显示无需压缩。（验证：在两种历史状态执行 `/compact`，检查请求类型、Memory 和中文反馈。）
- [ ] **C16（AC8）手动摘要输入保留 3K 余量。** 每个手动摘要或递归摘要请求的完整估算不超过窗口减 3K。（验证：捕获手动压缩产生的全部摘要请求，逐个检查估算上限。）
- [ ] **C17（AC9）近期原文与工具边界完整。** 压缩后近期区覆盖约 10K 且不少于 5 条；切点附近的工具调用和对应结果不会分离。（验证：构造切点正好落在多工具调用组内的历史，检查最终消息序列。）
- [ ] **C18（AC10）用户原文优先保留。** 预算允许时旧用户消息逐字不变；仅当用户原文本身阻止恢复安全预算时，才从最早消息开始归档并留下预览、意图和路径。（验证：运行“预算足够”和“用户原文超限”两个场景，比较最终用户内容与归档顺序。）
- [ ] **C19（AC11）正式摘要包含八个固定章节。** 缺任一章节或章节为空时不替换历史，完整结构才可提交。（验证：依次返回合法摘要和八种缺章节摘要，观察提交次数。）
- [ ] **C20（AC12）摘要调用没有工具能力且草稿被丢弃。** 请求不携带工具定义，提示明确禁止工具；最终 Memory、缓存、事件和日志均不包含分析草稿。（验证：捕获摘要请求和全部持久化/输出载体，搜索草稿唯一标记。）
- [ ] **C21（AC12）摘要模型尝试工具调用会失败。** 该次完整尝试计一次失败，不执行工具、不提交临时历史。（验证：让摘要模型返回工具调用，检查工具执行计数为 0、历史未变和失败计数加一。）
- [ ] **C22（AC13）全量超限时递归收缩后再正式摘要。** 系统优先处理最早可容纳块，临时摘要使工作副本逐步变小，最后仍产生一次覆盖全部收缩内容的正式摘要。（验证：捕获多层摘要请求，检查每层输入下降和最后请求覆盖范围。）
- [ ] **C23（AC13）单条超限消息先归档再分片摘要。** 每个分片请求均低于 3K 余量上限，临时摘要不进入最终历史，原消息可从路径完整恢复。（验证：运行单条超长用户消息场景，检查分片、最终历史和归档还原。）
- [ ] **C24（AC16）压缩边界消息始终存在。** 正式摘要和应急压缩后都明确提示需要具体代码或工具细节时重新读取，不得依据摘要猜测。（验证：检查两种压缩结果中摘要与近期原文之间的边界消息正文。）

## 重试、熔断与故障保护

- [ ] **C25（AC14）前两次失败、第三次成功只提交一次。** 三次均从同一原历史开始，失败事务回滚，第三次成功后 Memory 只替换一次且失败计数归零。（验证：脚本化三次摘要结果，记录输入历史、事务和 replace 调用。）
- [ ] **C26（AC15）连续三次失败后立即熔断并应急压缩。** 应急过程不调用 LLM，旧历史完整归档，本地索引、预览和边界替换历史，常规请求在低于自动安全线后继续。（验证：让三次摘要全部失败，检查 LLM 调用总数、熔断状态、归档和下一次常规请求。）
- [ ] **C27（AC15）熔断期间不再自动摘要。** 再次逼近上限时直接应急压缩；手动压缩成功或 `/clear` 后恢复自动摘要，手动失败则保持熔断。（验证：依次执行“再次超线 → 手动失败 → 手动成功 → clear”状态序列，观察每步调用类型和状态。）
- [ ] **C28（AC17）摘要失败类别均可观察。** API 错误、工具尝试、格式错误、正式摘要缺失、摘要过大、递归不收敛、超时和取消产生正确失败类别，且一次完整尝试只计一次。（验证：逐类注入故障，检查结构化报告、尝试次数和中文反馈。）
- [ ] **C29（AC17）归档写入失败时原文不丢且危险请求不发送。** Memory 保持原样，没有悬空路径，常规 LLM 调用数不增加，用户收到可操作错误。（验证：注入文件写入异常，比较前后历史、目录、请求和事件。）
- [ ] **C30（N10）取消和超时沿用现有运行约束。** 摘要调用同时服从 model timeout 与 run deadline，结束后无未关闭 stream、临时文件或半成品历史。（验证：分别让两个截止时间先到，检查错误类型和资源状态。）

## 协议与系统集成

- [ ] **C31（AC19）三种协议能归一化 usage 且缺失时兼容。** Chat、Responses 和 Anthropic 的合法 fixture 在 DONE 携带统一 usage；字段缺失或格式异常时对话仍正常完成并回退字符估算。（验证：运行三组合法/缺失/异常流式 fixture，比较事件序列和估算来源。）
- [ ] **C32（AC19）普通聊天与 thinking 语义不变。** 未触发压缩时，文本增量、thinking 展示、最终回复和 usage 事件顺序与 Stage 06 一致。（验证：运行既有普通聊天与 thinking 回归场景，比较事件序列。）
- [ ] **C33（AC19）工具、权限和取消语义不变。** 读写调度、HITL、plan-only、拒绝结果、取消和工具失败仍按原契约工作，压缩不会产生悬空 tool_call/result。（验证：运行既有 Agent、权限和 plan-only 回归测试，检查全部通过。）
- [ ] **C34（N7）压缩观测不泄漏正文。** 轻量、重量、递归、重试、熔断和应急均能看到动作、前后估算、归档数、尝试数和失败类别；报告与日志不包含归档正文或 API key。（验证：运行含唯一敏感标记的各类压缩场景，搜索所有事件和日志输出。）
- [ ] **C35（N6/N12）压缩实现边界保持集中。** Agent、Session、TUI 和协议层只做门面调用或数据适配，归档、估算、摘要、重试和熔断逻辑均位于独立 compact 包；关键预算、递归、路径和熔断路径有中文原因注释。（验证：检查依赖方向和关键路径注释，确认不存在 Agent 反向实现压缩策略。）
- [ ] **C36（F5）CLI 正确装配和回收资源。** 归档读取工具与普通工具共用调度链；正常退出和异常退出都关闭上下文缓存及 MCP 资源。（验证：运行 CLI 生命周期场景，记录工具注册与 close 顺序。）
- [ ] **C37（文档）示例和 README 与实际行为一致。** 三个示例均包含必填窗口配置，README 记录阈值、命令、缓存、重读、熔断和应急边界。（验证：加载全部示例配置并运行文档契约测试。）

## 端到端场景

- [ ] **C41（AC20）长任务正常压缩并重读。** 用户发起长编码任务，模型产生多个超大工具结果；系统先轻量归档，历史继续增长后自动结构化摘要，模型按边界提示分段重读旧材料并完成任务，全程没有上下文溢出。（验证：运行完整脚本化 CLI/Agent 场景，记录每次常规请求估算、压缩动作、重读结果和最终回复。）
- [ ] **C42（AC21）摘要三次失败后仍能继续并恢复。** 同一长任务中摘要连续失败三次，系统应急压缩后完成下一次常规请求；随后 `/compact` 成功，熔断被清除，后续再次接近上限时恢复自动摘要。（验证：运行故障恢复端到端场景，检查调用次数、状态转换和最终回复。）
- [ ] **C43（AC10/AC13）超长用户原文仍可恢复。** 用户发送单条超过摘要窗口的原文，系统归档、分片递归摘要并继续工作；模型随后可按路径完整重读原文。（验证：运行超长用户消息端到端场景，对比重读内容与原输入。）
