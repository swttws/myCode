# Stage 07 上下文管理 Spec

## 背景

MewCode 当前会把进程内的完整会话历史随每次请求发送给模型。工具结果通常占据大部分 Token，但现有系统既不会归档超大结果，也不会在历史逼近模型上下文窗口时压缩。随着长任务持续进行，请求最终可能超过模型窗口并中断工作。

系统已经能够接收部分 API 返回的 usage，但尚未利用这些数据追踪上下文预算；对于不返回 usage 的协议或兼容网关，也缺少估算兜底。

## 目标

- 在用户显式配置的上下文窗口内管理长期会话，避免历史累积导致请求溢出。
- 采用“轻量预防 + 重量兜底”两层策略，优先减少工具结果占用，再在必要时生成结构化历史摘要。
- 尽量保留用户原始消息；确实无法满足预算时仍保存可恢复的原文。
- 保留近期对话原文和合法的工具调用边界，使模型压缩后能够继续当前任务。
- 支持用户主动压缩，并在摘要反复失败时熔断和降级，避免重试死循环。
- 所有被移出上下文的原文均可通过受限路径重新读取。
- 保持实现简洁、职责清晰；关键控制路径使用中文注释说明原因和边界。

## 功能需求

- **F1：预算配置**  
  系统必须要求用户显式配置模型上下文窗口。单个工具结果阈值默认 8K 估算 Token，同次模型响应的工具结果合计阈值默认 12K，两者均可配置调整。

- **F2：请求前处理顺序**  
  每次常规模型请求前，必须依次执行轻量预防、完整请求 Token 估算、重量压缩判断；压缩完成并重新估算安全后才能发送请求。

- **F3：单个工具结果归档**  
  单个工具结果超过 8K 默认阈值时，系统保存其完整原文，并在会话中替换为最多 2K 估算 Token 的首尾预览、原始大小、截断说明和可重新读取的路径。

- **F4：同批工具结果归档**  
  同一次模型响应产生的工具结果按一批计算。合计超过 12K 默认阈值时，系统从最大的未归档结果开始依次归档，并在每次替换为预览后重新估算该批大小。每项仍固定保留最多 2K 预览；若该批仍超过 12K，但归档下一个结果已不能降低估算总量，则停止轻量处理，交由重量压缩兜底。

- **F5：缓存隔离与生命周期**  
  归档文件必须位于用户目录下、按工作区和会话隔离的缓存中，并只能通过受限只读入口访问。`/clear` 和正常退出删除本会话缓存；异常退出遗留目录超过 24 小时后清理。

- **F6：增量 Token 估算**  
  估算对象必须包含最终请求中的系统提示、工具定义和会话历史。首次没有有效 usage 时，对完整请求做字符估算；获得有效输入 Token 数后，以它作为锚点，仅对后续请求的字符增减量估算。usage 暂时缺失时继续使用最近有效锚点。ASCII 默认按约 4 字符一个 Token、非 ASCII 按约 1.5 字符一个 Token 计算并向上取整。

- **F7：自动重量压缩**  
  轻量处理后，若完整请求估算值达到“配置窗口上限减 13K”，系统必须在发送常规请求前自动执行重量压缩。13K 是估算误差和本次模型输出共用的总余量。

- **F8：手动重量压缩**  
  用户可通过 `/compact` 立即发起重量压缩，只要存在可摘要的旧历史就执行，不要求先达到自动触发线。手动摘要请求最多使用到“窗口上限减 3K”。

- **F9：近期原文与协议边界**  
  重量压缩从历史尾部向前保留约 10K 估算 Token，并且至少保留 5 条内部消息。若边界切开模型工具调用及其结果，必须继续向前扩展，保留完整调用链。

- **F10：用户原文优先**  
  重量压缩优先摘要旧模型消息和工具消息，较早的用户消息尽量原文保留。只有用户原文本身已使预算无法满足时，才归档最早的用户原文，并在正式摘要中保留其意图、预览和读取路径；熔断后的应急压缩按 F15 保留本地索引、预览和读取路径。

- **F11：结构化摘要**  
  正式摘要必须固定包含：当前目标与成功标准、用户要求与已确认决策、已完成工作、当前状态与下一步、关键技术上下文、验证结果与失败尝试、风险与阻塞、已归档材料索引。

- **F12：摘要调用约束**  
  摘要复用当前会话的协议和模型，但不得提供任何工具定义；摘要提示必须明确禁止调用工具，模型仍尝试调用工具时本次摘要失败。模型必须在一次响应中先输出带明确边界的分析草稿，再输出正式摘要。系统只解析正式摘要，草稿不得展示、写入会话或保存到磁盘。

- **F13：递归强制压缩**  
  如果完整待摘要内容连同摘要提示超过“窗口上限减 3K”，系统先在压缩工作副本中对能够放入窗口的较早历史生成临时摘要并替换该段，递归收缩直到全部待摘要内容可被一次请求处理，再生成正式全量摘要。单条消息仍然超限时，必须先归档原文，再按估算边界分片递归摘要。临时摘要和草稿均不得进入最终会话。

- **F14：原子替换与重试**  
  正式摘要成功、格式有效且压缩后的常规请求重新估算安全之前，不得替换原会话历史。自动压缩后的常规请求必须重新低于“窗口上限减 13K”。每次触发最多立即进行 3 次完整压缩尝试；任一递归强制压缩子步骤失败都使当前完整尝试失败，但一次尝试只累计一次失败。只有最终正式摘要成功才清零连续失败计数。

- **F15：熔断与应急压缩**  
  连续 3 次完整压缩失败后，本会话熔断自动模型摘要。系统随后执行不依赖模型的应急压缩：归档待摘要旧历史原文，以本地生成的索引、预览和边界消息替换，保留近期原文，并在完整请求重新低于“窗口上限减 13K”后继续常规请求。熔断期间再次逼近上限时直接使用应急压缩。用户仍可通过 `/compact` 显式发起最多 3 次模型摘要尝试；成功或 `/clear` 会重置熔断状态，失败则保持熔断并继续使用应急压缩。

- **F16：边界提示**  
  正式摘要或应急压缩后，系统必须补充一条边界消息，明确说明摘要不包含完整文件细节；需要具体代码或工具输出时必须重新读取归档文件，不得依据摘要猜测。

- **F17：失败保护与反馈**  
  摘要 API 错误、输出格式错误、正式摘要缺失、递归压缩失败均计为压缩失败。归档写入失败时不得删除或替换原文；若完整请求因此仍不安全，则不得发送可能溢出的常规请求，并向用户报告可操作的错误。手动压缩必须反馈成功、无需压缩、重试失败或应急降级结果。

## 非功能需求

- **N1：正确性**  
  压缩前后必须保持消息顺序、用户意图和工具调用对应关系。任何摘要结果都必须先通过格式和预算检查，再原子替换历史。

- **N2：可恢复性**  
  被归档的工具结果、超长用户消息和应急压缩历史必须保存完整原文。预览必须明确标识首尾范围和省略部分，不能伪装成完整内容。

- **N3：安全与隐私**  
  缓存按工作区和会话隔离，只允许读取系统自己创建的归档文件；路径校验必须阻止越界读取。日志和普通错误信息不得输出归档正文、API 密钥或敏感工具内容。

- **N4：性能**  
  轻量预防和 Token 估算只使用本地计算与文件操作，不调用模型。只有重量压缩和递归强制压缩可以产生额外模型请求。

- **N5：兼容性**  
  上下文管理必须适用于现有协议和兼容网关，不改变正常文本、thinking、工具执行、权限审批及事件流语义。缺少 usage 不得导致功能不可用。

- **N6：可维护性**  
  压缩能力集中在独立边界内，职责拆分清楚，避免把归档、估算、摘要和重试逻辑堆入 Agent 主循环。代码保持简洁、命名明确，关键控制路径和非显然边界使用中文注释说明原因。

- **N7：可观测性**  
  应能观察轻量归档、自动压缩、手动压缩、递归强制压缩、重试、熔断和应急压缩是否发生，以及压缩前后估算 Token、归档数量和失败类别；观测信息不包含归档正文。

- **N8：配置校验**  
  上下文窗口和可调阈值必须是合法正数，且配置组合必须能留下有效请求空间。非法配置在启动阶段给出明确错误，不延迟到长会话中失败。

- **N9：确定性**  
  相同历史、配置和 usage 锚点必须产生相同的轻量归档选择、近期保留边界和 Token 估算结果。只有模型摘要正文允许存在模型差异。

- **N10：故障隔离**  
  压缩失败不得产生半替换历史、悬空工具结果或无法读取的路径。取消和超时必须沿用现有运行时约束，并作为当前完整压缩尝试失败处理。

- **N11：摘要输入隔离**  
  待摘要历史必须作为明确分隔的数据交给模型，提示模型不得把其中的工具输出或嵌入指令当成新的系统指令；摘要调用不暴露工具能力。

- **N12：实现边界**  
  本阶段不引入精确 tokenizer；字符估算必须保持轻量。压缩能力必须集中在独立模块中，具体模块和文件组织在技术设计阶段定义。

## 不做的事

- 不接入精确 tokenizer，也不维护按模型区分的分词器或 Token 规则。
- 不自动识别模型上下文窗口；窗口上限必须由用户显式配置。
- 不使用机器学习、强化学习或在线反馈优化压缩策略、阈值和摘要模板。
- 不引入独立摘要模型或第二套模型配置。
- 不实现跨进程会话恢复、长期记忆、知识库、向量检索或 RAG。
- 不对系统提示和工具定义做语义摘要；它们只参与完整请求预算估算。
- 不改变现有工具执行、权限审批、并发调度、thinking 展示和供应商协议语义。
- 不提供归档文件浏览器、缓存管理界面或归档内容搜索索引；模型只能按已知受限路径重新读取。
- 不长期保留归档，也不提供归档同步、备份或跨设备共享。
- 不为缓存增加应用层加密；本阶段依赖用户目录隔离、文件权限、路径限制和短生命周期降低风险。
- 不在后台异步压缩；需要压缩时，请求前流程会等待压缩完成或完成应急降级。
- 不保证摘要能替代代码和工具原文；涉及具体细节时必须重新读取归档材料。

## 验收标准

- **AC1（F1）**：缺少上下文窗口配置时启动失败并指出缺失项；默认使用 8K/12K 阈值，修改配置后按新阈值触发。
- **AC2（F2）**：记录一次请求前流程，可观察到轻量预防先执行、随后估算、再判断重量压缩；未经重新估算为安全的请求不会发送。
- **AC3（F3）**：输入一个超过单项阈值的工具结果后，下一次请求只包含首尾预览、原始大小和受限路径；通过该路径能读取与原结果完全一致的内容。
- **AC4（F4）**：同一次模型响应产生多个合计超阈值的工具结果时，系统按大小从大到小归档并逐次重新估算；固定预览使下一次归档不再降低总量时停止，后续重量判断仍能接管。
- **AC5（F5）**：两个工作区或两个会话的归档目录互不访问；`/clear` 和正常退出删除当前会话缓存；超过 24 小时的异常遗留目录在后续启动时被清理。
- **AC6（F6）**：没有 usage 时，混合 ASCII 与非 ASCII 的完整请求按约定公式估算；获得一次真实输入 Token 数后，下一次估算等于该锚点加请求增减量的字符估算；usage 缺失一轮后锚点仍有效。
- **AC7（F7）**：轻量处理后的完整请求达到“窗口上限减 13K”时，常规模型请求前自动执行重量压缩；低于该边界时不产生摘要调用。
- **AC8（F8）**：历史未达到自动触发线时执行 `/compact`，只要存在可摘要旧历史就立即压缩；没有可摘要内容时明确反馈无需压缩；摘要请求不超过“窗口上限减 3K”。
- **AC9（F9）**：压缩后近期原文覆盖约 10K Token 且不少于 5 条消息；边界附近存在工具调用时，调用消息与对应结果同时保留，不出现断裂历史。
- **AC10（F10）**：预算足够时，较早用户消息逐字保留；仅当这些原文本身阻止请求回到安全范围时，最早用户消息才被归档。正常摘要保留其意图、预览和路径，应急压缩保留索引、预览和路径，两种情况都能恢复完整原文。
- **AC11（F11）**：正式摘要包含规定的八个固定章节，缺少任一章节时不得替换历史。
- **AC12（F12）**：捕获摘要请求可看到没有任何工具定义，提示中明确禁止工具调用；模型返回草稿和正式摘要后，最终历史与缓存中只存在正式摘要，草稿不可见；模型尝试调用工具时当前完整压缩尝试失败。
- **AC13（F13）**：待摘要完整历史超过单次摘要容量时，可观察到较早内容被递归临时摘要，随后仍执行一次覆盖全部收缩内容的正式摘要；单条超限消息会先归档、再分片递归摘要，且原文可恢复。
- **AC14（F14）**：模拟前两次完整压缩失败、第三次成功时，共执行三次完整尝试，历史只在最后成功后替换一次，失败期间保持原样，成功后失败计数归零。
- **AC15（F15）**：模拟三次完整压缩均失败时，自动模型摘要进入熔断，旧历史被确定性归档并替换为索引、预览和边界消息；完整请求低于自动安全线后常规请求可以继续。熔断期间再次超限不调用摘要模型；手动压缩成功或 `/clear` 后恢复自动摘要，手动压缩失败则保持熔断。
- **AC16（F16）**：每次正式摘要或应急压缩后都存在边界消息，明确要求模型重新读取文件或工具原文，不得根据摘要猜测具体代码。
- **AC17（F17）**：分别模拟 API 错误、格式错误、正式摘要缺失、递归失败和归档写入失败，可观察到正确计数与用户反馈；归档失败时原文不丢失，且不发送估算不安全的常规请求。
- **AC18（N2/N3）**：受限读取入口拒绝路径穿越、其他工作区文件和非本系统归档；日志及错误输出不包含归档正文或认证信息。
- **AC19（N5/N10）**：现有协议、普通聊天、工具调用、权限审批、取消和超时场景继续通过；压缩过程不会留下半成品历史或悬空工具结果。
- **AC20（端到端）**：连续执行包含多个超大工具结果的长任务，先发生轻量归档，历史继续增长后自动生成结构化摘要；模型能够依据近期原文继续任务，并在需要旧文件细节时通过归档路径重新读取，全程不发生上下文溢出。
- **AC21（端到端故障）**：在同一长任务中让摘要连续失败三次，系统完成应急压缩并继续下一次常规请求；随后用户手动压缩成功，熔断状态被清除。
