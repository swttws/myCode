# Stage 08 项目知识与长期记忆 Spec

## 背景

myCode 当前已经具备 Prompt Pipeline、Agent Loop、权限系统、MCP 工具接入和 Stage 07 上下文压缩能力。这些能力主要解决单次进程和单次会话内如何稳定工作：如何构造提示词、如何调工具、如何压缩过长上下文、如何在当前运行中保留近期历史。

但系统还不能在新会话启动后自动恢复项目知识、用户偏好和上次工作状态。用户每次重新启动或中断后继续使用时，Agent 仍然接近“失忆”状态：项目约定需要重复说明，用户偏好无法沉淀，历史会话不能可靠接续，自动总结出的长期知识也没有进入下一次请求。

Stage 08 要补上项目知识与长期记忆层。它通过项目指令文件、JSONL 会话存档和自动笔记三套机制，把工作记忆和长期记忆分层管理起来，并在每次处理用户请求前扫描和刷新，使 Agent 表现得像已经读过当前项目说明、最近会话和长期记忆。

## 目标

- 每次处理用户请求前扫描并刷新长期上下文。
- 自动恢复当前项目最近一个未过期会话。
- 项目级指令优先于用户级指令，高优先级内容先进入模型上下文。
- 会话只用 JSONL 追加写，不单独维护 meta 文件；会话 ID、标题、消息数、更新时间和可恢复状态都从 JSONL 扫描派生。
- 会话恢复能跳过坏行、截断不完整工具调用边界、在预算超限时先压缩、并插入必要的时间跨度提醒。
- 定期清理 30 天以上的过期会话。
- 自动笔记按用户偏好、纠正反馈、项目知识和参考资料分类，并分用户级和项目级存储。
- 用户偏好和纠正反馈默认进入用户级记忆；项目知识和参考资料默认进入项目级记忆；LLM 可以建议合法的归属调整和去重动作。
- 记忆索引在请求前注入上下文，使模型表现得像已经读过长期记忆。
- 新增目录和文件统一沿用 `mycode` 命名体系，不引入 `.mewcode` 命名。
- 保持 Stage 07 上下文压缩、现有 Agent 事件流、工具调度、权限审批和协议语义不变。
- 代码实现必须命名清晰，遵循现有 Python 项目风格；关键控制路径保留中文原因注释。

## 存储布局

- 项目指令文件位于 `mycode.md`、`.mycode/instructions.md` 和 `~/.mycode/instructions.md`。
- 会话存档位于 `~/.mycode/projects/<workspace_sha256>/sessions/<session_id>.jsonl`，每个会话一个 JSONL 文件。
- 用户级笔记位于 `~/.mycode/memory/`。
- 项目级笔记位于 `~/.mycode/projects/<workspace_sha256>/memory/`。
- 每条笔记文件使用带 frontmatter 的 Markdown 格式存储。
- 每个记忆目录包含一份索引文件，索引文件用于请求前注入上下文并受行数和大小限制。

## 功能需求

- **F1：请求前长期上下文刷新**  
  每次处理用户请求前，系统必须重新扫描项目指令、最近会话状态和记忆索引，并把可用结果注入本次模型上下文。刷新失败时不得阻断普通请求，除非失败会导致会话恢复不一致或路径安全边界失效。

- **F2：三层项目指令加载**  
  系统必须按固定三层加载手写 Markdown 指令：项目根 `mycode.md`、项目根 `.mycode/instructions.md`、用户目录 `~/.mycode/instructions.md`。项目级指令优先于用户级指令，高优先级内容必须排在更靠前的位置，让模型优先遵循。固定拼接顺序为项目根 `mycode.md`、项目根 `.mycode/instructions.md`、用户目录 `~/.mycode/instructions.md`。

- **F3：指令 include 安全处理**  
  项目指令必须支持 `@include` 引用其他 Markdown 文件。include 解析必须限制最大嵌套深度，用已访问集合防止环路，并拒绝跳出允许目录范围的路径。无法读取或被拒绝的 include 必须以诊断方式呈现，不得静默吞掉。

- **F4：会话 JSONL 追加存档**  
  会话必须以追加式 JSONL 保存，每条记录代表一个可恢复事件或消息。写入必须优先保证追加快、结构稳定、崩溃时最多丢失最后一行；系统不得为会话单独维护需要同步的 meta 文件。

- **F5：会话派生元数据**  
  需要会话 ID、标题、消息数、最近更新时间和可恢复状态时，系统必须通过扫描 JSONL 计算得到。会话 ID 使用 `YYYYMMDD-HHMMSS-xxxx` 形态，避免同秒创建冲突。

- **F6：最近会话自动恢复**  
  新会话启动后，系统默认自动选择当前项目最近一个未过期会话，并在第一次请求前恢复到可继续对话的历史状态。用户不需要显式输入会话 ID。

- **F7：恢复坏行容错**  
  恢复 JSONL 时，无法解析或结构无效的行必须跳过并记录诊断；坏行不得导致整个会话不可恢复。

- **F8：工具调用边界恢复**  
  恢复时如果发现模型工具调用没有配套工具结果，必须从该不完整边界截断历史，避免把悬空工具调用恢复给模型。

- **F9：恢复前预算保护**  
  恢复出的历史如果超过上下文预算，系统必须先触发现有压缩能力，把历史降到安全范围后再进入常规请求。不能安全压缩时，不得发送可能溢出的请求。

- **F10：时间跨度提醒**  
  恢复的会话距离上次活动间隔较久时，系统必须在上下文中插入一条时间跨度提醒，说明中断到恢复之间经过了多久，帮助模型避免误判“刚刚发生”。

- **F11：过期会话清理**  
  系统必须定期清理 30 天以上的过期会话。清理不得删除当前正在恢复或正在写入的会话。

- **F12：自动笔记分类**  
  自动笔记分为四类：用户偏好、纠正反馈、项目知识、参考资料。用户偏好和纠正反馈默认归入用户级；项目知识和参考资料默认归入项目级。LLM 可以基于内容建议调整归属和去重。

- **F13：自然停止后异步更新笔记**  
  每轮 Agent Loop 自然停止后，也就是模型最终回复且没有继续工具调用时，系统必须异步调用 LLM 更新自动笔记。该更新不得阻塞用户看到最终回复，也不得改变刚完成回合的事件顺序。

- **F14：笔记去重由 LLM 判断**  
  自动笔记更新时，系统必须把已有相关笔记摘要或索引提供给更新流程，由 LLM 判断新增、合并、修改或忽略。系统只负责格式、安全和写入边界，不用硬编码语义去重规则。

- **F15：用户级与项目级笔记隔离**  
  用户级笔记和项目级笔记必须分开存储、分开索引。项目级上下文不得覆盖用户级偏好；用户级偏好也不得伪装成项目事实。

- **F16：记忆索引注入**  
  请求前刷新时，系统必须把用户级和项目级记忆索引注入上下文，使模型表现得像已经读过这些长期记忆。索引必须控制在 200 行或 25KB 以内，以较先达到者为准。

- **F17：存储命名统一**  
  Stage 08 的新增持久化目录和文件统一沿用 `mycode` 命名体系，不引入 `.mewcode` 命名。

- **F18：明确不做高级检索**  
  本阶段不做向量数据库、RAG 检索、团队记忆同步或跨设备同步；长期记忆只通过文件、索引和请求前注入生效。

## 非功能需求

- **N1：安全性**  
  项目指令 include、会话恢复和笔记读取都必须有路径边界检查。项目级文件只能读取当前工作区允许范围内的内容；用户级文件只能读取 `~/.mycode` 下的约定文件。任何路径穿越、符号链接逃逸或越权读取都必须被拒绝。

- **N2：确定性**  
  相同文件内容、相同 JSONL 和相同索引状态，必须产生相同的指令拼接顺序、恢复历史、诊断列表和索引注入内容。只有自动笔记更新的 LLM 输出允许存在模型差异。

- **N3：可恢复性**  
  JSONL 会话存档必须优先保证崩溃后仍可最大程度恢复。坏行、半行和未完成工具边界只能影响局部历史，不得破坏整个会话文件。

- **N4：性能**  
  请求前扫描必须保持轻量：文件扫描、JSONL 读取和索引注入不调用模型。只有恢复预算超限压缩和自然停止后的自动笔记更新可以调用模型。

- **N5：隔离性**  
  用户级指令/笔记、项目级指令/笔记、会话存档和 Stage 07 的短期归档缓存必须有清晰边界。短期上下文归档不等同长期会话存档，不能混用生命周期规则。

- **N6：兼容性**  
  Stage 08 不得改变现有 OpenAI Chat、OpenAI Responses、Anthropic 协议语义，不得改变工具调度、权限审批、`plan-only`、`/compact`、`/clear` 和事件流的既有契约。

- **N7：可观测性**  
  系统必须能观察项目指令加载结果、include 诊断、恢复的会话 ID、跳过坏行数量、截断原因、注入的索引大小、自动笔记更新成功或失败。观测信息不得包含完整敏感正文、API key 或未授权文件内容。

- **N8：写入可靠性**  
  JSONL 追加、笔记 Markdown 写入和索引更新都必须避免留下不一致状态。索引更新应采用先写临时文件再原子替换；笔记写入失败不得破坏已有笔记。

- **N9：提示词边界**  
  项目指令、恢复提醒和记忆索引必须作为框架上下文注入，不能被误写入普通用户消息。它们应当影响模型行为，但不应伪装成用户本轮新请求。

- **N10：命名规范**  
  新模块、类、函数、文件和测试命名必须遵循现有 Python 项目风格，名称表达职责，不使用含糊缩写。涉及 Stage 08 的新包名和目录名必须统一使用 `mycode` 语义。

- **N11：注释规范**  
  代码注释必须聚焦关键原因和边界：include 路径校验、JSONL 崩溃恢复、工具调用截断、异步笔记写入、索引大小限制等位置应有中文注释。简单赋值、显而易见的流程不写空洞注释。

- **N12：测试可控性**  
  自动化测试必须使用 fake LLM、临时目录和固定时间源，不依赖真实网络、真实 API key 或用户本机已有的 `~/.mycode` 内容。

- **N13：实现边界**  
  本阶段只实现文件型长期记忆和会话恢复，不引入数据库、后台常驻进程、外部索引服务或新的供应商 SDK。

## 不做的事

- 不做向量数据库、embedding、语义 RAG、全文检索排序或相似度召回。
- 不做团队记忆、组织级共享记忆、跨用户同步或跨设备同步。
- 不做长期记忆加密、云端备份或远程存储。
- 不做会话管理 UI、会话列表面板、笔记浏览器或图形化编辑器。
- 不做自动迁移 `.mewcode` 目录；Stage 08 统一使用 `mycode` 命名体系。
- 不把 Stage 07 的短期归档缓存当作长期会话存档保存。
- 不要求真实 LLM 网络调用参与自动化验收；自动笔记更新用 fake LLM 或 fixture 覆盖。
- 不让自动笔记更新阻塞最终回复展示。
- 不做硬编码语义去重规则；语义去重和分类调整交给 LLM，系统只做格式、边界和写入安全。
- 不保证自动笔记一定提取出所有有价值信息；本阶段只保证触发、格式、隔离、去重流程和索引注入机制。
- 不支持用户手动指定会话 ID 恢复；默认恢复当前项目最近一个未过期会话。
- 不支持会话 meta sidecar 文件；标题、消息数、更新时间和状态都从 JSONL 扫描派生。
- 不改变现有权限系统语义，不新增对项目指令的“可信执行”能力；项目文件内容仍然只是模型上下文，不是可绕过权限的系统授权。

## 验收标准

- **AC1（F1）**：每次普通用户请求前都会重新扫描项目指令、最近会话和记忆索引；修改磁盘上的指令或索引后，下一次请求能观察到新内容生效。
- **AC2（F2）**：同时存在 `mycode.md`、`.mycode/instructions.md` 和 `~/.mycode/instructions.md` 时，最终上下文顺序为项目根指令、项目 `.mycode` 指令、用户指令。
- **AC3（F3）**：`@include` 能加载合法相对 Markdown；超过嵌套深度、循环 include、路径穿越、符号链接逃逸和不存在文件都会产生诊断，且不会读取越界内容。
- **AC4（F4/F5）**：发送多轮消息后，会话 JSONL 逐行追加；删除最后半行后仍能恢复前面完整记录；会话 ID、标题、消息数和更新时间都能通过扫描 JSONL 得到，不存在 meta 文件依赖。
- **AC5（F6）**：当前项目存在多个未过期会话时，新进程默认恢复最近更新时间最大的会话；过期会话不会被选中。
- **AC6（F7）**：JSONL 中混入坏行或结构错误行时，恢复跳过坏行并记录数量，其他有效消息正常恢复。
- **AC7（F8）**：JSONL 末尾存在 assistant tool call 但没有对应 tool result 时，恢复历史截断到该不完整工具边界之前，且不会向模型发送悬空工具调用。
- **AC8（F9）**：恢复历史超过预算时，常规请求前触发现有压缩流程；压缩成功后才发送请求，压缩失败且仍不安全时返回错误事件而不调用常规模型。
- **AC9（F10）**：恢复会话与上次活动间隔超过配置阈值时，本次请求上下文包含时间跨度提醒；间隔较短时不注入该提醒。
- **AC10（F11）**：创建超过 30 天的会话文件后触发清理，文件被删除；当前打开或最近恢复的会话不会被误删。
- **AC11（F12/F15）**：自动笔记更新把用户偏好/纠正反馈默认写入用户级目录，把项目知识/参考资料默认写入项目级目录；LLM 返回调整归属时按合法结果写入对应级别。
- **AC12（F13）**：模型最终回复且没有继续工具调用时，系统调度异步笔记更新；最终回复事件先返回给用户，笔记更新随后完成。
- **AC13（F14）**：已有笔记索引被提供给自动更新流程；fake LLM 返回新增、合并、修改和忽略四种动作时，文件结果符合动作语义。
- **AC14（F16）**：请求前注入的记忆索引同时包含用户级和项目级索引，且总量不超过 200 行和 25KB；超出时按确定性规则截断并产生诊断。
- **AC15（F17）**：新增目录和文件全部使用 `mycode` 命名体系；测试扫描不到 `.mewcode` 写入路径。
- **AC16（N1）**：项目指令 include、笔记读取和会话恢复的路径越界测试全部被拒绝，拒绝信息不包含目标文件敏感正文。
- **AC17（N5/N6）**：Stage 07 `/compact`、`/clear`、工具调用、权限审批、`plan-only` 和普通聊天回归测试继续通过。
- **AC18（N7）**：诊断或事件中可以看到加载文件数、include 错误数、恢复会话 ID、坏行数量、截断原因、索引大小和笔记更新结果，且不泄露 API key 或完整未授权正文。
- **AC19（N8）**：模拟笔记写入失败或索引替换失败时，已有笔记和旧索引保持不变；会话 JSONL 追加失败时本轮事件给出可观察错误。
- **AC20（N9）**：项目指令、时间跨度提醒和记忆索引只出现在请求构造的框架上下文中，不会作为普通用户消息写入 JSONL 会话历史。
- **AC21（N10/N11）**：新增代码命名清晰并符合现有风格；关键 include、JSONL 恢复、工具边界截断、异步笔记、索引限制路径均有中文原因注释。
- **AC22（端到端）**：用户在一个项目中连续对话、产生会话 JSONL 和自动笔记后，重启程序并发送新请求；系统自动恢复最近会话，注入项目指令和记忆索引，模型能接续之前的任务。
- **AC23（端到端故障）**：构造包含坏行、悬空工具调用、超预算历史和损坏索引的项目状态后发送请求；系统跳过坏行、截断悬空工具边界、压缩历史、降级索引注入，并给出诊断，过程中不发送不安全请求。
